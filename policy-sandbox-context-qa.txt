Guardian AI Q&A: Policy Graph, Sandbox, and Context Loop
Date: 2026-03-07

Q1:
"One more doubt. If the process goes through policy graph, isn't it being run already, then what's the use of sandbox. Also what abt context management, does the prompt going through the graph change the semantic meaning."

A1:
Policy graph and sandbox happen at different layers, not duplicates.

1) Policy graph is decision-time control:
- It checks whether a tool call should be allowed now.
- It reasons over sequence, risk nodes, transition rules, and exfiltration patterns.
- If blocked, execution never happens.
- If allowed, it means only that the call is policy-permitted.

2) Sandbox is execution-time control:
- It enforces how the tool runs: memory, timeout, file paths, network, and environment.
- It protects against buggy or malicious behavior even when policy approved the call.
- So: policy = authorization, sandbox = containment.

Analogy:
- Policy graph is the guard at the gate.
- Sandbox is the safety cage inside the lab.

Context management and semantics:
- Routing through the graph should not rewrite semantic intent by default.
- The graph should evaluate metadata about the call (tool name, params, sequence), not mutate user text.
- Behavior can still change indirectly via allow/deny/require-step decisions.
- That changes workflow, not linguistic meaning.

Best practice:
- Keep original prompt immutable in logs/session.
- Add separate normalized fields for policy checks.
- If transformation is needed, make it explicit and auditable with:
  - original_input
  - transformed_input
  - transformation_reason

Operational stack:
- Context manager tracks session/history.
- Policy graph decides permitted transitions.
- Sandbox constrains runtime effects.
- No silent meaning changes without traceability.


Q2:
"So we are following all these right?
Also decision-time vs execution-time, I don't think all the execution-time stuff are decided at decision-time, I think it must be a feedback kinda loop. So before knowing current execution, we can't make next decision. So how will wasm come here? will the session be created and called again and again?"

A2:
Short answer: yes in architecture, partially in current implementation details.

Your feedback-loop model is correct.

Current implementation status:
- Decision-time check: yes.
  - In src/tool_interceptor.cpp, validator->validate(...) runs first.
- Execution-time sandbox: yes.
  - Then sandbox_mgr->execute_tool(...) runs.
- Session context loop: yes.
  - Each call reads prior sequence via SessionManager::get_sequence(...), then appends current call via append_tool_call(...).

Runtime loop:
1) create_session() once.
2) For each tool call in that conversation: intercept(session_id, tool, params).
3) Interceptor reads history for that same session_id.
4) Policy decision is made.
5) If approved, sandbox executes (Wasm or mock runtime).
6) Session is updated.
7) Next call uses updated session history.

Where Wasm fits:
- Wasm is in execution-time layer (SandboxManager/WasmExecutor).
- Policy graph does not execute tools; it gates whether execution is allowed.
- If WasmEdge is unavailable, mock runtime path is used.

Important nuance in current code:
- In src/tool_interceptor.cpp, approved calls are appended to session even if sandbox execution fails.
- That can make next decision treat an execution attempt as progression.

On semantic meaning:
- Graph checks should not alter semantic meaning.
- Current path validates tool_name plus prior sequence, and sends params JSON to sandbox.
- No prompt-rewriting layer is shown in this flow.
- So behavior changes via allow/block sequencing, not by rewriting user intent.

Suggested tighter loop options:
1) Append only on sandbox success.
2) Store call status (approved/executed/failed), and validate against executed calls only.
