Guardian AI: Product Positioning, Runtime Scope, and Use Cases
Date: 2026-03-07

Q1) If a tool cannot compile to Wasm, what happens? What are we compiling?
- We compile only tool execution logic to .wasm, not the whole Guardian app.
- Guardian control-plane/runtime components can remain native C++.
- Good Wasm candidates:
  1) Deterministic utility tasks (parsing, transforms, checks, report generation)
  2) Data processing with controlled I/O
  3) Risky helper tools that should run with restricted capabilities
- Poor Wasm candidates:
  1) Tools requiring broad OS/kernel access, privileged networking, drivers, GPU-heavy native deps
  2) Legacy binaries tightly coupled to host libraries
- If not Wasm-compatible:
  1) Keep native with stricter policy checks, allowlists, and audit
  2) Use container isolation (Docker/K8s)
  3) Use hybrid model: Wasm first, container fallback

Q2) Docker vs WasmEdge, and how sandboxing works here
- Docker/K8s: OS-level isolation (namespaces/cgroups), broader compatibility, heavier startup.
- WasmEdge: runtime-level isolation for Wasm modules, lower startup overhead, capability-focused controls.
- In this project, sandbox controls map to:
  1) memory_limit_mb
  2) timeout_ms
  3) allowed_paths
  4) network_access
  5) controlled environment variables
- Recommended architecture:
  1) WasmEdge for compatible tools
  2) Container fallback for incompatible tools
  3) Unified policy and audit stream across both

Q3) Can we integrate NeMo Guardrails?
- Yes. Best positioning is complementary:
  - NeMo: conversation/LLM flow guardrails
  - Guardian: runtime/tool execution guardrails
- Integration chain:
  1) NeMo decides whether a tool should be called
  2) Guardian enforces whether it may execute now under policy
  3) Sandbox executes with telemetry and audit

Q4) Better use cases than trivial demos
- Strong flagship scenario: enterprise VPN/data egress guardrails.
- Example:
  1) Sensitive source tools: repo/data/PII access
  2) External destination tools: public email/external HTTP/upload
  3) Rule: sensitive -> external requires redact/tokenize/approval path
  4) Runtime block if egress is outside corporate VPN/CIDR policy

Q5) CLI vs GUI
- Keep CLI for engineering/testing.
- Make GUI primary and necessarily for demos/adoption:
  1) Live agent timeline (request -> policy decision -> sandbox result)
  2) Animated graph traversal (allowed/blocked transitions)
  3) Session replay and incident drill-down
  4) Explainability panel (violated rule + safe alternatives)

Q6) Which organizations can use this?
- Enterprises running internal AI copilots in regulated contexts.
- SaaS platforms exposing agentic workflows on customer data.
- Security/compliance-heavy industries (finance, healthcare, defense, telecom).
- Internal champions: security engineering, AI platform, compliance/risk, dev platform teams.

Q7) Can we show this through codebase usage?
- Yes. Add an enterprise developer-assistant flow example:
  1) read_code allowed
  2) deploy_production blocked without approval
  3) approval_request allowed
  4) deploy_production allowed post-approval
- This demonstrates policy + runtime enforcement in a practical workflow.

Q8) What exactly is the product and how is it used?
- Product components:
  1) Policy control plane (author/version policies)
  2) Runtime enforcement SDK/gateway (intercept tool calls)
  3) Sandbox runner (WasmEdge-first, container fallback)
  4) Audit/incident UI (replay, explainability, compliance export)
- Typical use:
  1) App team integrates SDK
  2) Security team publishes policy packs
  3) Runtime enforces each tool call
  4) SOC/compliance consumes audit artifacts
- Demo idea with virtual third party:
  - Example org: Northstar Health + external vendor QuickMail AI
  - Simulate PHI exfil attempt -> block -> suggest safe path -> approve after redaction/approval

Follow-up on Q1: Product positioning and runtime claim

What should we do now?
- Market as "policy-driven tool execution security" with "Wasm-first sandboxing".
- Do not claim universal runtime for any OS syscall workload.
- Document clear compatibility tiers now and container fallback roadmap.

Recommended messaging (safe and accurate)
- "Guardian secures agent tool execution with policy validation and sandboxed runtime controls."
- "Best for deterministic, auditable tool workloads; supports Wasm-native execution and planned container isolation for broader compatibility."

Should Docker/K8s be future scope?
- Yes, if timeline is tight. Keep as clearly documented roadmap:
  1) Phase A: Wasm-first runtime (current)
  2) Phase B: Docker/K8s runner fallback
  3) Phase C: policy parity + unified telemetry across runtimes

Other OS virtualization options
- Firecracker microVMs (strong isolation, more ops complexity)
- gVisor/Kata Containers (container hardening approaches)
- Native process sandboxing per OS (Windows Job Objects, Linux seccomp/AppArmor) as interim controls
- For this project stage, Wasm + planned container fallback is the best complexity/security tradeoff.

"Instead of saying runtime for any OS, say it sandboxes your own OS functions; for Linux syscalls run on Ubuntu."
- This direction is much better and more credible.
- Suggested refined statement:
  - "Guardian enforces policy and sandbox controls on tool execution. OS-specific capabilities depend on the host runner. Linux-specific behavior is guaranteed on Linux runners (or Linux containers)."
- This avoids overpromising and aligns with engineering reality.

- Position Guardian as execution-policy and enforcement infrastructure, not a universal syscall abstraction layer.
- Be explicit about current support matrix and planned fallback runtimes.
- Lead demos with high-value enterprise exfiltration and approval workflows, not toy scenarios.
